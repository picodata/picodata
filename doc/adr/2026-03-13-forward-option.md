# Возможность ограничения количества сетевых пересылок во время исполнения запроса

## Мотивация

В настоящий момент при получении SQL запроса Пикодата вычисляет множество
бакетов, на которых должен выполниться запрос, и рассылает его соответствующим
репликасетам. Возможна ситуация, когда бакеты содержатся в репликасете, на узел
которого пришел запрос. В таком случае запрос будет выполнен на текущем узле или
отправлен на реплику внутри репликасета. Этот факт широко используется в "умных"
драйверах, что позволяет улучшить производительность за счет уменьшения сетевых
пересылок. Драйвер заранее вычисляет множество бакетов для запроса и отправляет
его на соответствующий репликасет. Подробнее с идеей можно ознакомиться в
[**этом**](https://git.picodata.io/core/picodata/-/blob/984b6a76a3b9a75c6ee4803ecaa99b3e879c4338/doc/adr/2025-09-12-pgproto-query-metadata.md)
ADR.

В рамках текущего АДР предлагается дать возможность пользователям ограничивать
количество пересылок по сети во время исполнения DQL и DML запросов. Так, если
для исполнения запроса требуется больше пересылок, чем разрешено, то
пользователь получит ошибку на этапе исполнения запроса.

## Решение

### Опция `forward`

Предлагается добавить опцию **forward**(см. задачу
[#2722](https://git.picodata.io/core/picodata/-/issues/2722)), которая может
иметь три различных значения:

1. **on** - бакеты, на которых будет исполняться запрос, принадлежат разным
   узлам; в таком случае запрос будет выполнен через scatter-gather по лидерам
   затронутых репликасетов.

2. **ro_to_rw** - все бакеты запроса должны принадлежать **одному узлу**, но при
   исполнении _допустима пересылка_ с координатора запроса на лидера
   соответствующего бакетам репликасета. **Иначе кидаем ошибку.**

3. **off** - все бакеты должны принадлежать **одному узлу**, а клиент обязан сам
   отправить запрос на лидера соответствующего бакету репликасета. **Иначе
   кидаем ошибку.**

Иными словами, мы имеем нарастающую степень локальности: 
1. локальности нет ни в каком смысле;
2. по крайней мере, все бакеты запроса принадлежат одному узлу;
3. истинная локальность (клиент отправил запрос на узел, хранящий бакеты).

Значение опции можно будет указывать в части `OPTION` SQL запроса по аналогии с
уже существующими `sql_vdbe_max_row`, `sql_vdbe_opcode_max` и `read_replica`.
Например:

```sql
SELECT * FROM t WHERE a = 1 OPTION (FORWARD = OFF);
```

#### Взаимодействие опции `forward` с опцией `read_preference`

В настоящий момент Пикодата позволяет для DQL запросов указывать опцию
`read_preference`, с помощью которой можно выбрать на каких узлах будет
исполняться запрос. Подробнее с опцией можно ознакомиться в
[**документации**](https://docs.picodata.io/picodata/devel/reference/sql/dql/#params).

Ниже приведена таблица всех возможных комбинаций двух опций. Галочкой отмечены
разрешенные комбинации. В остальных случаях пользователь получит ошибку с
объяснением, почему комбинация невалидна.


| `forward` / `read_preference` |  `on`  | `off`  | `ro_to_rw` |
|:-----------------------------:|:------:|:------:|:----------:|
|            `any`              |   ☑    |   ✖    |     ☑      |
|           `leader`            |   ☑    |   ☑    |     ☑      |
|           `replica`           |   ☑    |   ✖    |     ☑      |

В случае `read_preference = any / replica` запрос может быть выполнен на
реплике, что противоречит `forward = off`. При указании такой комбинации
пользователь получит ошибку. Таким образом, опция `forward` имеет приоритет над
`read_preference`.


### Динамическая проверка выполнимости опции `forward`

Осуществлять проверку выполнимости опции Пикодата будет во время выполнения
запроса, т.е. динамически. Бакеты, вычисленные для каждого поддерева(motion
узла), будут проверяться на соответствие опции `forward`. Если вычисленные
бакеты не соответсвуют указанной опции, то Пикодата возвращает ошибку. Например,
в случае когда указана опция `forward = off`, а вычисленные бакеты указывают на
то, что запрос будет выполнен как scatter-gather по лидерам репликасетов,
Пикодата вернет пользователю ошибку.

Рассмотрим описанное выше на примере. Пусть есть таблица

```sql
CREATE TABLE t (a INT PRIMARY KEY, b INT);
```

и запрос

```sql
INSERT INTO t VALUES (1 + 1, 2) OPTION (FORWARD = OFF);
```

Рассмотрим `explain` этого запроса:
```sql
                                QUERY PLAN                                 
---------------------------------------------------------------------------
 insert "t" on conflict: fail
     motion [policy: segment([ref("COLUMN_1")]), program: ReshardIfNeeded]
         values
             value row (data=ROW(1::int + 1::int, 2::int))
 execution options:
     sql_vdbe_opcode_max = 45000
     sql_motion_row_max = 5000
 buckets = unknown
(8 rows)
```

Данный запрос будет исполняться в 2 стадии. Сначала Пикодата вычислит бакеты,
соответствующие поддереву `motion` узла и проверит их на соответствие указанной
опции `forward`. Если условия выполнены, то Пикодата продолжит исполнять запрос,
иначе выбросит ошибку. Далее аналогичная проверка будет произведена для самого
верхнего узла, т.е. `insert`. Важно отметить, что хоть опция и проверяется во
время исполнения запроса, она **не может** привести к ситуации, когда часть DML
выполнилась, а часть нет. **Т.е. невозможна ситуация "разорванного" DML**.

### Статический анализ допустимых опций для `forward`

Вместе с этим предлагается дать возможность пользователю узнать, с какими
значениями опции forward запрос гарантированно сможет выполниться. Эта
функциональность будет добавлена в вывод explain вне зависимости от указания
опции в запросе. Например:

```sql
admin=> explain select * from t where a = $1 \bind 1 \g
                      QUERY PLAN                       
-------------------------------------------------------
 projection ("t"."a"::int -> "a", "t"."b"::int -> "b")
     selection "t"."a"::int = 1::int
         scan "t"
 execution options:
     sql_vdbe_opcode_max = 45000
     sql_motion_row_max = 5000
 guaranteed forward options:
     forward = on
     forward = ro_to_rw
     forward = off
 buckets = [1934]
(7 rows)
```

Анализ допустимых опций будет производиться, т.е. статически без выполнения
самого запроса. Важно отметить, что анализ будет выполнен на этапе **`Bind`**
после подстановки параметров. 

Рассмотрим конкретный пример. Пусть есть таблицы

```sql
CREATE TABLE t (a INT PRIMARY KEY, b INT);
CREATE TABLE g (a INT PRIMARY KEY, b INT) DISTRIBUTED GLOBALLY;
```

и запрос

```sql
INSERT INTO t SELECT * FROM g;
```

На данный момент Пикодата не может статически рассчитать бакеты, поэтому для
запроса единственной доступной опцией `forward` будет являться только `on`. Вывод
explain для этого запроса:
```sql
admin=> explain insert into t select * from g;
                             QUERY PLAN                             
--------------------------------------------------------------------
 insert "t" on conflict: fail
     motion [policy: segment([ref("a")]), program: ReshardIfNeeded]
         projection ("g"."a"::int -> "a", "g"."b"::int -> "b")
             scan "g"
 execution options:
     sql_vdbe_opcode_max = 45000
     sql_motion_row_max = 5000
 guaranteed forward options:
     forward = on
 buckets = unknown
(8 rows)
```

С другой стороны, возможна ситуация, когда бакеты, вычисленные из строк таблицы
`g`, будут находиться на одном узле. В таком случае запрос может быть выполнен с
более строгими ограничениями.

Иными словами, вывод допустимых опций в explain является `best-effort` попыткой
указать пользователю, с какими опциями запрос гарантированно может быть
выполнен.

### Примеры использования опции `forward`

В данном разделе рассматриваются применения опции `forward` во время работы с
Пикодатой. В каждом подразделе будем подразумевать, что запросы используют
таблицу t:

```sql
CREATE TABLE t (a INT PRIMARY KEY, b INT);
```

Здесь и далее мы будем скрывать вывод `explain`, не относящийся к информации об
опции `forward` и бакетах запроса. Позднее это также будет доступно через
указание дополнительных фасетов `buckets` и `forward` в команде `explain`.

#### `INSERT` запросы

Рассмотрим запрос:
```sql
INSERT INTO t VALUES ($1, $2), ($3, $4) \bind 1 2 3 4 \g
```

Пусть его `explain` выглядит так:
```sql
                                QUERY PLAN                                 
---------------------------------------------------------------------------
 guaranteed forward options:
     forward = on
     forward = ro_to_rw
     forward = off
 buckets = [1934,1958]
(5 rows)
```

Из этого сразу следует, что запрос может быть **гарантированно** выполнен с
опцией `forward = off`. Это может быть полезно для случая "умной" вставки
данных, когда нужно удостовериться в том, что данные заливаются правильно.

#### `SELECT` запросы

Рассмотрим запрос:
```sql
SELECT * FROM t WHERE a = $1 or a = $2 \bind 1 2 \g
```

Его `explain`:
```sql
                            QUERY PLAN                            
------------------------------------------------------------------
 guaranteed forward options:
     forward = on
     forward = ro_to_rw
 buckets = [1410,1934]
(4 rows)
```

После этого пользователь может выполнить запрос с опцией `forward = ro_to_rw`,
в результате чего будет совершено не более одной сетевой пересылки.

Аналогично опция применима для запросов `UPDATE` и `DELETE`.
